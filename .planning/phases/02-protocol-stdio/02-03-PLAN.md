  <action>
Create a test MCP server that can be spawned as a subprocess for integration testing.

1. **Option A (preferred): Use in-process transport for testing**

   rmcp supports in-memory transports for testing. Instead of spawning a subprocess, create paired transports:
   ```rust
   // In test, create a pair of in-memory transports
   let (client_transport, server_transport) = rmcp::transport::io::duplex(64);
   ```

   If rmcp doesn't provide a duplex/pair transport, use **tokio::io::duplex()** to create paired streams and wrap them.

2. **Option B (fallback): Create a test binary**

   Add to `crates/tl-mcp/Cargo.toml`:
   ```toml
   [[bin]]
   name = "tl-test-mcp-server"
   path = "tests/test_server.rs"
   ```

   Create `tests/test_server.rs`:
   ```rust
   //! Test MCP server for integration testing.
   //! Exposes two tools: "echo" (returns input) and "add" (adds two numbers).

   use tl_mcp::server::{TlServerHandler, ToolDef, serve_stdio};
   use std::sync::Arc;

   fn main() {
       let handler = TlServerHandler::builder()
           .name("test-server")
           .version("1.0.0")
           .tool(ToolDef {
               name: "echo".to_string(),
               description: "Returns the input unchanged".to_string(),
               input_schema: serde_json::json!({
                   "type": "object",
                   "properties": {
                       "message": { "type": "string" }
                   },
                   "required": ["message"]
               }),
               handler: Arc::new(|args| {
                   let msg = args.get("message")
                       .and_then(|v| v.as_str())
                       .unwrap_or("no message");
                   Ok(serde_json::json!({ "echoed": msg }))
               }),
           })
           .tool(ToolDef {
               name: "add".to_string(),
               description: "Adds two numbers".to_string(),
               input_schema: serde_json::json!({
                   "type": "object",
                   "properties": {
                       "a": { "type": "number" },
                       "b": { "type": "number" }
                   },
                   "required": ["a", "b"]
               }),
               handler: Arc::new(|args| {
                   let a = args.get("a").and_then(|v| v.as_f64()).unwrap_or(0.0);
                   let b = args.get("b").and_then(|v| v.as_f64()).unwrap_or(0.0);
                   Ok(serde_json::json!({ "result": a + b }))
               }),
           })
           .build();

       serve_stdio(handler).expect("Server failed");
   }
   ```

Try Option A first. If in-memory transport isn't available, use Option B.

IMPORTANT: The test server must be a standalone binary OR use in-memory transports. It cannot share the test process's stdin/stdout.

<task type="auto">
  <name>Task 2: Write integration tests for full MCP lifecycle</name>
  <files>crates/tl-mcp/tests/integration.rs</files>
  <action>
Create `crates/tl-mcp/tests/integration.rs` with comprehensive tests:

1. **test_client_server_handshake** — Connect client to server, verify:
   - Connection succeeds (no error from connect())
   - server_info() returns Some with correct name and version
   - Server capabilities include tools

2. **test_list_tools** — After connecting:
   - list_tools() returns 2 tools (echo, add)
   - Tools have correct names, descriptions, and input schemas

3. **test_call_tool_echo** — Call the echo tool:
   - call_tool("echo", {"message": "hello"}) returns {"echoed": "hello"}
   - Result is not marked as error

4. **test_call_tool_add** — Call the add tool:
   - call_tool("add", {"a": 2, "b": 3}) returns {"result": 5}

5. **test_call_nonexistent_tool** — Call a tool that doesn't exist:
   - call_tool("nonexistent", {}) returns an error (ToolError or ProtocolError)

6. **test_ping** — After connecting:
   - ping() returns Ok(())

7. **test_disconnect_cleanup** — Verify clean shutdown:
   - Connect to server
   - disconnect() returns Ok(())
   - is_connected() returns false

8. **test_security_policy_blocks** — With sandbox SecurityPolicy:
   - connect() with sandboxed policy returns PermissionDenied error
   - No subprocess is spawned

If using in-memory transport, tests can run directly. If using test binary, use:
```rust
let client = McpClient::connect(
    &test_server_path(),
    &[],
    None, // no security policy
)?;
```

Where test_server_path() returns the path to the compiled test binary.

IMPORTANT:
- Each test must clean up (drop client, wait for process)
- Use #[tokio::test] if needed, or std::sync blocking
- Set reasonable timeouts to prevent test hangs
